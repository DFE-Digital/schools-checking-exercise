Design changes

Change: 'Your details' now includes a section giving the user information if they need to change their details.

Reason: Currently 'Your details' seems like a reference page with no call-to-action. Users shouldn't step through this page if there is no action to take.

Severity: Low

Change: Changed table structure for 'Your pupil details'. Table has a title and Name and UPN have moved to become column headings.

Reason: Currently 'Student name' is repeated in every row, this change avoids repetition throughout the table. This takes up less space and is easier for screen readers.

Severity: Low

Change: Pupil UPN has been added as a column in 'Your pupil details'.

Reason: UPN is referenced throughout the journey, especially when pupils need merged. If the user needs to reference this page it is useful to give them all pupil details needed throughout the journey.

Severity: Medium

Change: Pupil details are in alphabetical order (by last name).

Reason: There should be structure to the names. It's an assumption that if students need merged the names would be similar, so this could make it easier to spot. Another assumption is that the users would know pupils by name first, before their UPN.

Severity: Medium

Change: Header now has links to 'Your details', 'Your pupil details',' 'DfE account' and 'Sign out'.

Reason: Gives the user the option to access reference pages at any point during their journey. They can also access their account that they've signed in with, and they are able to sign out.

Severity: Medium

Change: Changed the 'Create a request' page to a question, 'Do you want to request to remove a pupil?'

Reason: Has the same behaviour as the current form, however is more clear and provides a call-to-action.

Severity: High

Change: When the user does not want to make a request, the structure of the summary page has changed. The radio buttons are removed in favour of a summary table where the user can change that specific answer.

Reason: Currently the radio buttons don't have an impact. The change uses a standard Design System pattern - Check answers (design-system.service.gov.uk).

Severity: High

Change: 'Who do you want to remove?' pupil selection is now an accessible autocomplete - Accessible autocomplete - Alpha Gov (github.com).

Reason: Some schools will have a lot of pupils, this gives them a way to find pupils through their name or UPN while still maintaining accessibility.

Severity: High

Change: Evidence upload page uses the File upload component - File upload (design-system.service.gov.uk).

Reason: Keeps the service in line with the Design System.

Severity: Medium

Flow changes

Change: Users can access another version of the 'Your details' page from the header. This version does not have a section giving information to change their details.

Reason: This can be used as a reference point and is accessible at any point during their journey.

Severity: Low

Change: Users can access another version of the 'Your pupil details' page from the header. This version has less information.

Reason: This can be used as a reference point and is accessible at any point during their journey.

Severity: Medium

Change: Check your answers before submitting request or uploading evidence follow the Check answers pattern - Check answers (design-system.service.gov.uk). The user can change specific answers from these screens.

Reason: This can be used as a reference point and is accessible at any point during their journey.

Severity: High

Change: Evidence upload has been restructured. Text areas that are included on the pages have been removed. If the text area is required (e.g for Not on roll), this has been added as a question to the journey.

Reason: This keeps the one question a page structure. This means the user will have less chance of timing out when uploading evidence if there is a required answer. The text areas aren't always relevant to uploading files, so this can minimise users assuming they're connected.

Severity: High

Recommendations

Applies to: Flow of the pages

Recommendation: Skip this page in the journey and let the user access their pupil details through the header. Flow would be

  • 'Your details', then
  • 'Do you want to request to remove a pupil?'

Reason to change: 'Your pupil details' can be over 100 pupils long. This page is currently just used as a reference point so it's not necessary for the user to land on this.
